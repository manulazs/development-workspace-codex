# Lessons - 2026-05

## Repository Is Not Runtime State

- Date: 2026-05-19
- Status: validated and promoted to policy
- Context: Public-template review of the Codex development workspace.
- Symptom: Earlier docs treated repository health and local runtime synchronization as closely coupled.
- Cause: The repository started as an operational workspace and retained local-runtime language after it became a reusable public template.
- Validated fix: Separate repository validation from optional profile-based runtime adoption. Healthchecks validate the repository; installers preview or copy selected adoption profiles only when requested.
- Evidence: `workspace-manifest.json`, profile-based installers, and updated healthchecks now operate without inspecting local `~/.codex` state.
- Reuse rule: Public repository health must never depend on a maintainer's private runtime. Runtime validation, if needed, must be a separate opt-in helper.
- Risk: Documentation can drift back toward personal environment assumptions unless audits explicitly check portability.

## Validator Dependencies Must Be Repository-Relevant

- Date: 2026-05-19
- Status: validated
- Context: Skill and agent validation during workspace governance work.
- Symptom: A validator can fail or be skipped because a local Python environment lacks a dependency.
- Cause: Repository checks and local runtime/toolchain checks were mixed together.
- Validated fix: Keep lightweight repository validation inside the healthcheck and report optional validator availability clearly.
- Evidence: Healthchecks now validate skill frontmatter and agent TOML directly, and run repository-local validators when available.
- Reuse rule: Do not report full validation success unless the validator and its dependencies actually ran.
- Risk: Lightweight validation checks structure only; semantic validation still needs tool-specific checks when changing a capability.
