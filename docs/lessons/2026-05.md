# Lessons - 2026-05

## Workspace Repo Is Not The Runtime Install

- Date: 2026-05-19
- Status: validated
- Context: Global audit of the Codex development workspace.
- Symptom: The repository listed many skills and custom agents, but the active `~/.codex` profile did not contain matching installed agents and only had a small skill set.
- Cause: The repository is a reproducible source of truth, while `~/.codex` is a separate runtime installation target.
- Validated fix: Add `scripts/healthcheck.ps1` to detect repo-to-runtime drift and `scripts/install-workspace.ps1` to copy skills and agents explicitly.
- Evidence: Runtime inspection showed repo capabilities and active profile capabilities were different before implementation.
- Reuse rule: Before relying on a repo skill or custom agent, run healthcheck or inspect `~/.codex`.
- Risk: Do not auto-install or delete runtime files without explicit approval or a safe preview.

## Validator Dependencies Must Be Explicit

- Date: 2026-05-19
- Status: validated
- Context: Skill validation during workspace audit.
- Symptom: `quick_validate.py` failed because `PyYAML` was not available in the active Python environment.
- Cause: The validation rule depended on an undeclared Python package.
- Validated fix: Healthcheck now reports the dependency gap clearly and performs lightweight built-in frontmatter validation.
- Evidence: The failed validator produced `ModuleNotFoundError: No module named 'yaml'`.
- Reuse rule: Do not report full skill validation success unless the validator and its dependencies actually ran.
- Risk: Lightweight validation checks structure only; it does not replace full semantic validation.
